# 餐饮日报周报自动化 Skill 规范

> 本文件是「便宜坊马连道餐厅经营数据自动化」项目的 **Skill 规范（单一事实源之一）**。
> 目的：让 Codex / Claude Code / 其他 Agent 下次进入仓库后，**不需要用户重新解释业务规则**，即可理解本项目的输入、处理规则、输出、验收标准与禁止事项。
>
> 配套文档：
> - 项目背景与记忆：`PROJECT_MEMORY.md`
> - 智能体接入规范与禁止事项：`docs/AGENT_ONBOARDING.md`
> - 完整业务流程：`docs/WORKFLOWS.md`
> - 日期/口径治理：`docs/date_and_metric_policy.md`
> - 字段字典与表结构：`docs/data_schema.md`
> - 进入项目自动读取：`AGENTS.md`、`CLAUDE.md`

---

## 1. Skill 定位

将门店每日经营截图，自动转化为：

1. **结构化经营数据**（落库到历史与富字段事实表）
2. **日报**（AI 诊断 + 经营建议）
3. **周报**（自然周维度聚合 + 诊断）
4. **可视化看板**（日报/周报 HTML 与 PNG 长图）

并自动**推送到飞书群**，形成「截图 → 结构化 → 日报/周报 → 看板 → 飞书」的经营数据闭环。

本 Skill 是 CaiHub 餐饮经营数据闭环的早期模块，不是单纯的「图片转 Excel」工具。

---

## 2. 适用场景

- 门店每日经营截图处理（PNG / JPG）
- 日报生成与飞书推送
- 周报生成与飞书推送（自然周，周一触发）
- 周报可视化长图 / HTML 看板生成与推送
- 后续扩展：月报、多门店对比、节气/天气/活动建议、周边客群建议、菜品销量分析

---

## 3. 输入

| 输入项 | 说明 | 来源 |
|--------|------|------|
| 门店经营截图 | 当日日报截图（左侧营业收入表 + 右侧销售日报表） | 放入监听目录 `/Users/ming/Restaurant/daily-input/马连道` |
| 门店名称 | 例如 `便宜坊马连道` | 参数 / 配置默认值 |
| 图片表头日期 | **business_date 的唯一来源**，由读图识别 | 图片表头 |
| 经营数据字段 | 收入、客流、客单价、折扣率、占比、烤鸭销量等 | 读图识别 |
| 可选配置文件 | `field_map.yaml`、`data/holiday_calendar_cn.json` 等 | 仓库内 |
| 飞书机器人配置 | `FEISHU_WEBHOOK` / `FEISHU_APP_ID` / `FEISHU_APP_SECRET` | `.env`（**不打印、不提交**） |

> processing_date（系统处理日期）可作为可选输入或日志参考，但**绝不能覆盖 business_date**。

## 4. 核心数据字段

当前 `data/store_history.csv` 标准字段（V1 骨架，只追加，不覆盖、不删除）：

| 字段 | 含义 |
|------|------|
| `date` | 业务日期（business_date，来自图片表头） |
| `store_name` | 门店名称 |
| `revenue` | 当日营业额 |
| `customer_count` | 客流（来客数） |
| `avg_ticket` | 客单价 |
| `month_yoy` | 月累计同比 |
| `discount_rate` | 折扣率 |
| `dine_in_ratio` | 堂食占比 |
| `takeaway_ratio` | 外卖占比 |
| `roast_duck_sales` | 烤鸭销量 |
| `warning_level` | 经营等级（健康 / 警示 / 异常） |
| `summary` | AI 摘要 |
| `suggestions` | AI 经营建议 |

富字段事实表 `data/daily_facts.csv`（附加层，**不动 V1 骨架**）：在以上基础上增加日期维度（business_month、business_week_start/end、month_to_date、previous_month_mtd、previous_week_same_weekday 等）与口径拆分字段（gross/net revenue、dine_in/takeaway/online revenue、source_image_hash 等）。详见 `docs/data_schema.md`。

## 5. 日报业务规则（强约束）

- **business_date 必须来自图片表头识别出的日期。**
- 禁止使用系统日期、文件创建日期、监听时间作为业务日期。
- processing_date 可以存在，但**只能用于日志**，不进入业务字段。
- 如果图片表头日期识别失败，**流程必须中止**，不允许 fallback 到系统日期。
- 如果传入参数 `--date` 与图片表头日期不一致，**必须以图片表头日期为准**，并记录 `warning_processing_date_differs` 警告。
- 不允许为了补齐日报而伪造日期或修改图片中的真实业务日期。
- 表头日期 ≠ business_date 的入库尝试将被**硬阻止**（绝不把 05-31 截图写成 06-01）。
- 同一 store + business_date 默认禁止覆盖；更正需 `mode='amend' + reason`，自动备份 + 审计。
- `source_image_hash`（sha256）用于防重复截图。

---

## 6. 周报业务规则（强约束）

- 周报周期为**自然周：周一至周日**。
- 周一收到周日数据后，**先完成周日日报**，再触发上一自然周周报。
- 周报触发**不是固定 9 点**，而是在周日数据处理完成后触发。
- 不使用 crontab；周六日报完成不触发周报。
- 如果中间缺少某天数据，可以在周报中**醒目标记缺失日期**，但**不允许补造数据**。
- 周报与周报可视化看板应**一起推送飞书**。
- 周报数据来源**必须来自已落库的历史数据**（`data/store_history.csv` 真实存在的日报日期），**不是重新解释截图**。
- 同一个自然周周期**只推送一次**，通过 `data/weekly_state.json` 防重复。

---

## 7. 可视化看板标准

- 覆盖两类看板：**日报看板** 与 **周报看板**。
- **HTML 优先**，必要时导出 **PNG 长图**（周报默认走融合版高清长图）。
- **文字不能重叠**，数据展示要完整、清晰。
- 看板应包含：收入、客流、客单价、折扣率、堂食/外卖占比、烤鸭销量、趋势、异常预警与建议。
- 图片尺寸可**根据内容自适应**，不必死限制固定尺寸。
- 看板只读取已验证的周报/日报数据，**不修改任何业务数据或业务日期**；字段缺失时显示「暂无」或隐藏模块，不报错、不伪造。

---

## 8. 飞书推送标准

- **日报推送**：互动卡片（KPI + 诊断等级 + 建议）；分析图在凭证可用时上传。
- **周报推送**：周报卡片 + 周报看板图片一起推送。
- 推送前**必须完成本地校验**（日期连号检查、字段完整性、敏感信息检查）。
- 推送**失败必须记录日志**。
- **不允许重复推送同一业务日期**，除非用户明确要求重推（`is_already_pushed` 守卫按 `pipeline_log.csv` 的 `date` 列命中即跳过）。
- 推送过程**不打印 / 不写入** webhook、token、app secret、image_key 等敏感值。

---

## 9. 安全与权限边界（不可违反）

- 不读取、不打印、不提交 `.env`。
- 不暴露 webhook、token、app secret、image_key。
- 不自动 `git push`，除非用户明确要求。
- 不自动修改历史 CSV（`store_history.csv`、`pipeline_log.csv` 只追加/定位修改目标行，不覆盖、不删除）。
- 不自动删除历史输出文件。
- 不在没有确认的情况下覆盖飞书正式文档。
- **所有涉及历史数据修正的操作必须先询问用户**，并先备份 + 记录到 `data/data_correction_log.md`。
- 禁止 `git add .` / `git add -A`；禁止 `cat` 全量 `pipeline_log.csv`（用 grep/tail）。
- 含真实营业数字的文件（`store_history.csv`、`*.bak`、`data_correction_log.md` 等）排除出 git。

## 10. Agent 执行顺序（适合 Codex 直接执行）

1. 读取 `PROJECT_MEMORY.md` / `AGENTS.md` / `CLAUDE.md` / `README.md`。
2. 检查 `data/pipeline_state.json`（当前应处理哪一天、上次是否已推送、下一步动作）。
3. 确认输入截图（监听目录中的新图片）。
4. 识别图片表头日期（business_date）。
5. 校验业务日期（表头识别失败则中止；`--date` 不一致则以表头为准并告警）。
6. 抽取经营数据（左侧营业收入表 + 右侧销售日报表，重复字段加大类前缀）。
7. 写入历史数据（`store_history.csv` 追加；富字段写入 `daily_facts.csv`，防污染校验）。
8. 生成日报（Excel + report JSON + AI 诊断）。
9. 生成日报看板（4 张分析图 / HTML）。
10. 推送飞书（推送前安全检查，避免重复推送）。
11. 判断是否触发周报（仅周一且前一天为周日时触发）。
12. 如触发：生成周报 + 周报看板，并一起推送飞书。
13. 记录日志（`pipeline_log.csv`、`pipeline_state.json`、`logs/watch_daily_folder.log`），按规则 git commit（push 需用户确认或既定自动化授权）。

---

## 11. 验收标准

- 日期正确：business_date == 图片表头日期，未被系统/文件/监听日期覆盖。
- 字段完整：核心字段齐全，缺失项标注而非伪造。
- 无敏感信息泄露：日志/输出/提交均无 `.env`、webhook、token、app secret。
- 日报推送成功：飞书收到日报卡片（及凭证可用时的分析图）。
- 周报触发逻辑正确：仅在周一处理完周日数据后触发，且自然周只推一次。
- 看板无文字重叠，数据完整清晰。
- 日志完整：状态文件与运行日志均正确记录。
- 不污染历史数据：未覆盖/删除历史 CSV，未把某天截图写成另一天。

## 13. 环境变量与运营上下文（附加层，2026-06-05 新增）

运营上下文是叠加在日报之上的**附加层**：由 `main._build_ops_context(business_date)` 产出，
经 `daily.context` 注入日报 JSON → 落库 `daily_facts.csv` → 进 AI 诊断建议。
**任何失败都不阻断 V1 主流程；缺数据时诚实记 `no_data` / `暂无`，绝不伪造。**

### 13.1 节气（确定性）
- 单一真相源：`data/solar_terms_cn.json`（权威发布日期），经 `solar_terms.py` / `date_dimension.py` 派生。
- 禁止用寿星公式等近似推断（个别年份 ±1 天误差）；表外年份记 `no_data`。
- 用途：临近节气做时令营销（如临近夏至/小暑推时令套餐）；当天恰逢节气可做节气主题营销。

### 13.2 天气（高德，可降级）
- 由 `weather_amap.py` 调高德天气 API（`restapi.amap.com`）。免费版无历史天气。
- `business_date` 通常是过去日期 → 当日天气记"暂无"，**绝不把采集日天气当成业务日天气**；
  仅把采集时刻实况 + 预报作为弱参考，预报主要用于"明日/近期经营建议"。
- 用途示例：预报雨天 → 主推外卖/线上、堂食备等位茶歇；高温 → 主推精酿/凉菜；降温 → 主推热菜/烤鸭暖锅。

### 13.3 相关环境变量（.env，不打印、不提交真实值）

| 变量 | 默认 | 说明 |
|------|------|------|
| WEATHER_ENABLED | true | 是否启用天气附加层 |
| WEATHER_PROVIDER | amap | 天气 provider（目前支持 amap） |
| WEATHER_API_KEY | 空 | 高德 Web 服务 key；未配置则天气全部"暂无"，不伪造、不报错 |
| WEATHER_CITY_ADCODE | 110102 | 城市 adcode（默认北京西城区，马连道所在） |
| WEATHER_CITY_NAME | 北京市西城区 | 城市名（仅展示用） |
| WEATHER_TIMEOUT | 8 | 高德接口超时秒数 |

### 13.4 未来可扩展的运营上下文（暂未实现，需有真实数据源才接入）
- **门店活动 / 预订状态**：本地促销活动、当日宴会预订量，可显著影响客流与备货。
- **人员状态**：当班人手、是否缺岗，影响出餐与服务建议。
- **周边客群 / 商圈**：门店位置周边写字楼/社区客群结构。
> ⚠️ 以上任何一项都必须有真实数据来源才入库；无来源时不得编造，沿用"暂无"口径。
